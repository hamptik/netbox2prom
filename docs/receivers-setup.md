## 1. Prometheus (SNMP Exporter)

The Prometheus generator writes one YAML file per group into the `scrape_dir` directory. Each file contains a fully-formed `scrape_configs` entry targeting the SNMP exporter.

### Output file structure

For a group named `snmpv3_network_huawei-automated`, the generator writes:

```
<scrape_dir>/
├── snmpv3_network_huawei-automated.yml
├── snmpv3_ipmi_idrac-automated.yml
├── snmpv1_ups_eaton-automated.yml
└── ...
```

Each file looks like:

```yaml
scrape_configs:
  - job_name: snmpv3_network_huawei-automated
    scrape_interval: 4m
    scrape_timeout: 2m
    metrics_path: /snmp
    params:
      auth:
        - public_v3_network_auth_aes
      module:
        - system
        - ip_mib
        - huawei
    static_configs:
      - targets:
          - 10.10.10.2
        labels:
          instance: sw-core-01
          criticality: high
      - targets:
          - 10.10.10.3
        labels:
          instance: sw-core-02
          criticality: medium
    relabel_configs:
      - source_labels: [__address__]
        target_label: __param_target
        action: replace
      - target_label: __address__
        replacement: snmp-exporter:9116
        action: replace
      - source_labels: [__param_target]
        target_label: node_ip
        action: replace
      - target_label: device_type
        replacement: network
        action: replace
      - target_label: vendor
        replacement: huawei
        action: replace
```

**How relabeling works:**

| Step | What it does |
|---|---|
| `__address__` → `__param_target` | Copies the target IP into the `target` query parameter for snmp_exporter |
| `__address__` → `snmp-exporter:9116` | Rewrites the scrape address to point at the SNMP exporter proxy |
| `__param_target` → `node_ip` | Preserves the original IP as a `node_ip` label for dashboards/alerts |
| `device_type` | Added if the group config has `device_type` set (e.g., `network`, `ups`, `storage`) |
| `vendor` | Added if the group config has `vendor` set |

> Groups with **zero** matching devices result in the output file being **deleted** (if it previously existed). This keeps the scrape directory clean — no empty jobs.

### prometheus.yml configuration

Point Prometheus at the generated files using `scrape_config_files`:

```yaml
# prometheus.yml
scrape_config_files:
  - /opt/configs/prometheus-scrape/*.yml
```

> `scrape_config_files` is supported in Prometheus 2.43+. It automatically picks up new files and removes deleted ones on reload.

You can combine this with your existing static scrape configs:

```yaml
global:
  scrape_interval: 5m

scrape_configs:
  - job_name: prometheus
    static_configs:
      - targets: ['localhost:9090']

scrape_config_files:
  - /opt/configs/prometheus-scrape/*.yml
```

### SNMP Exporter setup

The `params` in each generated group reference **auth profiles** and **modules** that must exist in your `snmp.yml` (the SNMP exporter's configuration file):

```yaml
# snmp.yml (snmp_exporter config)
auths:
  public_v3_network_auth_aes:
    version: 3
    security_level: authPriv
    username: monitor
    authentication_protocol: SHA
    authentication_passphrase: "your-auth-pass"
    privacy_protocol: AES
    privacy_passphrase: "your-priv-pass"

  public_v2_network_auth:
    version: 2
    community: public

  public_v1_ups_auth:
    version: 1
    community: public

modules:
  system:
    walk:
      - sysUpTime
      - sysName
      - sysDescr
      # ...

  ip_mib:
    walk:
      - ipAdEntAddr
      # ...

  huawei:
    walk:
      - 1.3.6.1.4.1.2011
      # ...

  dell_idrac:
    walk:
      - 1.3.6.1.4.1.674.10892
      # ...
```

> The `auth` and `module` names in your `config.yml` groups must match the keys in `snmp.yml`. The generator passes them through verbatim.

### Hot-reload

`netbox2prom` can trigger a Prometheus config reload after writing files. Configure the reload endpoint:

```yaml
prometheus:
  reload_address: "http://prometheus:9090"
```

The service sends `POST http://prometheus:9090/-/reload` after each generation cycle.

**Requirements:**

- Prometheus must be started with `--web.enable-lifecycle`:
  ```bash
  prometheus --web.enable-lifecycle --config.file=prometheus.yml
  ```
- Or in Docker:
  ```yaml
  services:
    prometheus:
      image: prom/prometheus
      command:
        - '--web.enable-lifecycle'
        - '--config.file=/etc/prometheus/prometheus.yml'
      ports:
        - "9090:9090"
  ```

To disable reload (e.g., when relying on file-watchers), leave `reload_address` empty:

```yaml
prometheus:
  reload_address: ""
```

### Docker volumes

The generator writes to `scrape_dir` (default: `/opt/configs/prometheus-scrape`). Mount this as a shared volume so Prometheus can read it:

```yaml
# docker-compose.yml
services:
  netbox2prom:
    volumes:
      - prometheus_scrape:/opt/configs/prometheus-scrape

  prometheus:
    volumes:
      - prometheus_scrape:/opt/configs/prometheus-scrape:ro

volumes:
  prometheus_scrape:
```

## 2. Grafana Alloy (Blackbox ICMP)

The Alloy generator produces a single JSON file containing all ICMP targets, grouped by the rules in your `config.yml`.

### Output file structure

The output is a JSON array of `{ targets, labels }` objects — the same format used by Prometheus file-based service discovery:

```json
[
    {
        "targets": [
            "10.10.10.5"
        ],
        "labels": {
            "__param_module": "icmp",
            "environment": "prod",
            "request": "icmp",
            "criticality": "high",
            "name": "srv-web-01",
            "os": "linux",
            "alloy": "true",
            "node_ip": "10.10.10.5",
            "device": "device"
        }
    },
    {
        "targets": [
            "10.10.10.100"
        ],
        "labels": {
            "__param_module": "icmp",
            "environment": "prod",
            "request": "icmp",
            "criticality": "",
            "name": "sw-access-05-kvm",
            "alloy": "false",
            "node_ip": "10.10.10.100",
            "device": "kvm"
        }
    }
]
```

Key behaviors:

- `default_labels` from the config are applied to every target; per-group `labels` override them.
- Setting a label to `null` in a group **removes** it for that group's targets.
- The `__param_module` label is special — it tells the blackbox exporter which prober module to use.

### Alloy configuration

Create the following components in your Alloy configuration file (e.g., `/etc/alloy/config.alloy`):

```alloy
// Read the JSON targets file generated by netbox2prom
discovery.file "icmp" {
    files           = ["/etc/alloy/blackbox_targets.json"]
    refresh_interval = "30s"
}

// Run the blackbox exporter as an inner component
prometheus.exporter.blackbox "icmp" {
    config_file = "/etc/alloy/blackbox_modules.yml"
    targets     = discovery.file.icmp.targets
}

// Relabel: expose the target IP as 'instance' and the name as 'target_name'
discovery.relabel "icmp" {
    targets = prometheus.exporter.blackbox.icmp.targets

    // Use the target IP as the 'instance' label
    rule {
        source_labels = ["__param_target"]
        target_label  = "instance"
    }

    // Promote the 'name' label to 'target_name'
    rule {
        source_labels = ["name"]
        target_label  = "target_name"
    }
}

// Scrape the blackbox exporter and forward to Prometheus
prometheus.scrape "blackbox" {
    targets    = discovery.relabel.icmp.output
    forward_to = [prometheus.remote_write.default.receiver]
}

// Remote write (adjust to your endpoint)
prometheus.remote_write "default" {
    endpoint {
        url = "http://prometheus:9090/api/v1/write"
    }
}
```

### Blackbox module file

Create `/etc/alloy/blackbox_modules.yml` with the ICMP prober configuration:

```yaml
modules:
  icmp:
    prober: icmp
    timeout: 5s
    icmp:
      preferred_ip_protocol: ip4
```

> The module name (`icmp`) must match the `__param_module` label value in the generated targets.

### Hot-reload

`netbox2prom` can trigger an Alloy config reload after writing the targets file. Configure the reload endpoint:

```yaml
probe_icmp:
  reload_address: "http://alloy:12345"
```

The service sends `POST http://alloy:12345/-/reload` after each generation cycle.

**Requirements:**

- Alloy's HTTP server must be listening (enabled by default on port `12345`):
  ```bash
  alloy run --server.http.listen-addr=0.0.0.0:12345 config.alloy
  ```
- Or in Docker:
  ```yaml
  services:
    alloy:
      image: grafana/alloy:latest
      command:
        - run
        - --server.http.listen-addr=0.0.0.0:12345
        - /etc/alloy/config.alloy
      ports:
        - "12345:12345"
  ```

To disable reload (e.g., when relying on `discovery.file`'s `refresh_interval` to pick up changes), leave `reload_address` empty:

```yaml
probe_icmp:
  reload_address: ""
```

### Docker volumes

Mount the targets file so both Alloy and netbox2prom can access it:

```yaml
# docker-compose.yml
services:
  netbox2prom:
    volumes:
      - alloy_targets:/etc/alloy

  alloy:
    volumes:
      - alloy_targets:/etc/alloy:ro
      - ./blackbox_modules.yml:/etc/alloy/blackbox_modules.yml:ro

volumes:
  alloy_targets:
```

## 3. Grafana Alloy (Blackbox HTTP)

The `probe_http` generator produces a JSON file containing HTTP targets derived from NetBox IPAM services that have a URL in the `website` custom field.

### Output file structure

The output is a JSON array of `{ targets, labels }` objects — same format as ICMP:

```json
[
    {
        "targets": [
            "https://wiki.example.com"
        ],
        "labels": {
            "__param_module": "http_2xx",
            "environment": "prod",
            "request": "http",
            "name": "wiki.example.com",
            "service_name": "wiki.example.com",
            "device": "web"
        }
    }
]
```

Key behaviors:

- The target address is the URL from the service's `website` custom field.
- `__param_module` controls which blackbox module is used (default: `http_2xx`). Can be overridden per group.
- The `name` label is derived from the URL hostname by default (configurable via `name_field`).

### Alloy HTTP configuration

Add HTTP discovery components alongside your existing ICMP config:

```alloy
// Read the JSON targets file generated by netbox2prom (probe_http)
discovery.file "http" {
    files           = ["/etc/alloy/probe_http_targets.json"]
    refresh_interval = "30s"
}

// Run the blackbox exporter for HTTP checks
prometheus.exporter.blackbox "http" {
    config_file = "/etc/alloy/blackbox_modules.yml"
    targets     = discovery.relabel.http_in.output
}

// Relabel: set instance from the service name
discovery.relabel "http_in" {
    targets = discovery.file.http.targets

    rule {
        source_labels = ["name"]
        target_label  = "instance"
    }

    rule {
        source_labels = ["name"]
        target_label  = "target_name"
    }
}

// Final relabel: expose the URL as 'address', service name as 'instance'
discovery.relabel "http" {
    targets = prometheus.exporter.blackbox.http.targets

    rule {
        source_labels = ["__param_target"]
        target_label  = "address"
    }

    rule {
        source_labels = ["target_name"]
        target_label  = "instance"
    }

    rule {
        action = "labeldrop"
        regex  = "job"
    }
}

// Scrape the blackbox exporter and forward to remote write
prometheus.scrape "blackbox_http" {
    targets    = discovery.relabel.http.output
    forward_to = [prometheus.remote_write.default.receiver]
    job_name   = "integrations/blackbox_http"
}
```

> The blackbox module file must include an `http_2xx` module (or whatever module name you use in `__param_module`):

```yaml
# /etc/alloy/blackbox_modules.yml
modules:
  icmp:
    prober: icmp
    timeout: 5s
    icmp:
      preferred_ip_protocol: ip4

  http_2xx:
    prober: http
    timeout: 10s
    http:
      preferred_ip_protocol: ip4
      valid_http_versions: ["HTTP/1.1", "HTTP/2.0"]
      valid_status_codes: [200]
```

### Hot-reload

Same as ICMP — configure the reload endpoint:

```yaml
probe_http:
  reload_address: "http://alloy:12345"
```

### Docker volumes

Mount the HTTP targets file alongside the ICMP targets:

```yaml
# docker-compose.yml
services:
  netbox2prom:
    volumes:
      - alloy_targets:/etc/alloy

  alloy:
    volumes:
      - alloy_targets:/etc/alloy:ro
      - ./blackbox_modules.yml:/etc/alloy/blackbox_modules.yml:ro

volumes:
  alloy_targets:
```

## 4. Grafana Alloy (Blackbox TCP)

The `probe_tcp` generator produces a JSON file containing TCP port targets derived from NetBox IPAM services. These are services with the monitoring tag that **do not** have a `website` field (those are handled by `probe_http`), but do have a TCP protocol, ports, and IP addresses.

### Output file structure

The output is a JSON array of `{ targets, labels }` objects — same format as ICMP and HTTP:

```json
[
    {
        "targets": [
            "10.15.5.7:22"
        ],
        "labels": {
            "__param_module": "tcp_connect",
            "environment": "prod",
            "request": "tcp",
            "name": "oxt-vs-pbs01:22",
            "service_name": "SSH2",
            "device": "service"
        }
    }
]
```

Key behaviors:

- The target address is `<ip>:<port>`, where the IP comes from the service's `ipaddresses` field and the port from `ports`.
- If a service has multiple ports, a separate target entry is generated for each port.
- `__param_module` controls which blackbox module is used (default: `tcp_connect`).
- The `name` label defaults to `<device_name>:<port>` (e.g., `oxt-vs-pbs01:22`).

### Alloy TCP configuration

Add TCP discovery components alongside your existing ICMP and HTTP config:

```alloy
// Read the JSON targets file generated by netbox2prom (probe_tcp)
discovery.file "tcp" {
    files           = ["/etc/alloy/probe_tcp_targets.json"]
    refresh_interval = "30s"
}

// Run the blackbox exporter for TCP checks
prometheus.exporter.blackbox "tcp" {
    config_file = "/etc/alloy/blackbox_modules.yml"
    targets     = discovery.relabel.tcp_in.output
}

// Relabel: set instance from the service name
discovery.relabel "tcp_in" {
    targets = discovery.file.tcp.targets

    rule {
        source_labels = ["name"]
        target_label  = "instance"
    }

    rule {
        source_labels = ["name"]
        target_label  = "target_name"
    }
}

// Final relabel: expose the target address, service name as 'instance'
discovery.relabel "tcp" {
    targets = prometheus.exporter.blackbox.tcp.targets

    rule {
        source_labels = ["__param_target"]
        target_label  = "address"
    }

    rule {
        source_labels = ["target_name"]
        target_label  = "instance"
    }

    rule {
        action = "labeldrop"
        regex  = "job"
    }
}

// Scrape the blackbox exporter and forward to remote write
prometheus.scrape "blackbox_tcp" {
    targets    = discovery.relabel.tcp.output
    forward_to = [prometheus.remote_write.default.receiver]
    job_name   = "integrations/blackbox_tcp"
}
```

### Blackbox module file (TCP)

The blackbox module file must include a `tcp_connect` module (or whatever module name you use in `__param_module`):

```yaml
# /etc/alloy/blackbox_modules.yml
modules:
  icmp:
    prober: icmp
    timeout: 5s
    icmp:
      preferred_ip_protocol: ip4

  http_2xx:
    prober: http
    timeout: 10s
    http:
      preferred_ip_protocol: ip4
      valid_http_versions: ["HTTP/1.1", "HTTP/2.0"]
      valid_status_codes: [200]

  tcp_connect:
    prober: tcp
    timeout: 5s
```

> The `tcp_connect` module performs a simple TCP connect check — it opens a connection to the target and verifies it succeeds. No application-level handshake is performed.

### Hot-reload

Same as ICMP and HTTP — configure the reload endpoint:

```yaml
probe_tcp:
  reload_address: "http://alloy:12345"
```

> If `probe_icmp`, `probe_http`, and `probe_tcp` all share the same `reload_address`, Alloy is reloaded only **once** per cycle.

### Docker volumes

Mount the TCP targets file alongside the other target files:

```yaml
# docker-compose.yml
services:
  netbox2prom:
    volumes:
      - alloy_targets:/etc/alloy

  alloy:
    volumes:
      - alloy_targets:/etc/alloy:ro
      - ./blackbox_modules.yml:/etc/alloy/blackbox_modules.yml:ro

volumes:
  alloy_targets:
```

## 5. syslog-ng (Hostname Rewrites)

The syslog generator produces **rewrite rules** that fix or override the `PROGRAM`/`HOST` fields for devices that send malformed syslog hostnames (common with network switches).

### How it works

1. The generator iterates over syslog groups in your `config.yml`.
2. For each matching device, it instantiates the group's `template` with the device's name and IP.
3. All generated rules are assembled into a single `rewrite` block.
4. The block is written to a **fully managed file** (`config_file`) — the entire file is generated from scratch on each run. No in-place editing of existing syslog-ng configs.
5. If the file content changed, syslog-ng is reloaded automatically via `syslog-ng-ctl reload`.

**Safety layers:**

| Layer | Description |
|---|---|
| **Structural validation** | Python-level check: brace balance (string-aware), block structure |
| **Syntax check** | `syslog-ng --syntax-only` on the temp file before applying (if binary available) |
| **Atomic write** | Temp file + `os.replace()` — no partial writes on crash |
| **Health check + rollback** | After reload, verifies syslog-ng is still running. If not — restores the previous config from `.bak` and reloads again |

### Output structure

Given this config:

```yaml
syslog:
  config_file: /etc/syslog-ng/conf.d/netbox2prom.conf
  block_name: fix_hostnames
  control_socket: /var/lib/syslog-ng/syslog-ng.ctl
  epilogue_template: |-
        set("${{SOURCEIP}}", value("HOST")
            condition("${{HOST}}" eq ""));
  groups:
    huawei_hpe_switches:
      conditions:
        role: switch
        vendor: [huawei, hpe]
        snmp_ver: any_except
        snmp_ver_exclude: [0]
      host_field: main_ip
      template: |-
            set("{name}", value("PROGRAM") \
                condition("${{HOST}}" eq "{ip}"));
```

The generated `/etc/syslog-ng/conf.d/netbox2prom.conf` will contain:

```syslog-ng
# Generated by netbox2prom - do not edit manually.
# This file is fully managed and will be overwritten on the next run.
rewrite fix_hostnames {
    set("sw-core-01", value("PROGRAM") \
        condition("${HOST}" eq "10.10.10.2"));
    set("sw-access-05", value("PROGRAM") \
        condition("${HOST}" eq "10.10.10.3"));
    set("${SOURCEIP}", value("HOST")
        condition("${HOST}" eq ""));
};
```

> If no devices match any group, the managed file is **removed** and syslog-ng is reloaded to pick up the change.

### Template syntax

Each group defines a `template` string that is instantiated per matching device.

**Available placeholders:**

| Placeholder | Value |
|---|---|
| `{name}` | Device name, cleaned (whitespace trimmed, `"` escaped to `\"`) |
| `{ip}` | IP address from `host_field` (default: `main_ip`) |

**Escaping syslog-ng variables:**

syslog-ng uses `${VAR}` syntax for its own macros. Since the template uses Python `.format()`, you must **double the braces** to produce literal braces in the output:

```
${{HOST}}     →  ${HOST}       (in the output file)
${{SOURCEIP}} →  ${SOURCEIP}   (in the output file)
```

The `epilogue_template` is instantiated **without** device context (no `{name}` / `{ip}` substitution), so it's useful for rules that apply globally — like a fallback to set `HOST` from the source IP when it's empty.

### Applying the rewrite block

The generated file is placed in the syslog-ng `conf.d` directory. Include it from your main syslog-ng configuration. Edit `/etc/syslog-ng/syslog-ng.conf`:

```syslog-ng
# Include configs from conf.d (including the netbox2prom-managed file)
@include "/etc/syslog-ng/conf.d/*.conf"

source s_network {
    udp();
    tcp();
};

log {
    source(s_network);

    # Apply the rewrite rules generated by netbox2prom
    rewrite(fix_hostnames);

    # Forward to your destination
    destination(d_messages);
};
```

> The `@include` directive must come **before** the `log {}` block that references `rewrite(fix_hostnames)`, because syslog-ng processes the config top-to-bottom.

### Automatic reload

After writing the config file, netbox2prom reloads syslog-ng via the **control socket**:

```
syslog-ng-ctl reload --control=<control_socket>
```

The control socket path is configured via `syslog.control_socket` in `config.yml` (default: `/var/lib/syslog-ng/syslog-ng.ctl`).

**Post-reload health check:**

1. Wait a few seconds (`health_check_delay`, default 3s).
2. Run `syslog-ng-ctl status` to verify syslog-ng is still running.
3. If syslog-ng is **not** running — restore the previous config from `.bak` and reload again.
4. If syslog-ng **is** running — clean up the `.bak` file.

If `syslog-ng-ctl` is not available (e.g., running outside a container without syslog-ng installed), the reload is skipped with a warning.

### Docker setup

When running in Docker, two things must be shared between the netbox2prom and syslog-ng containers:

1. **Config directory** (`/etc/syslog-ng/conf.d`) — so netbox2prom can write the generated config.
2. **Control socket** (`/var/lib/syslog-ng`) — so netbox2prom can run `syslog-ng-ctl reload`.

**UID/GID matching:** The control socket has restrictive permissions. The netbox2prom container must run with the same UID/GID as syslog-ng, or a UID/GID that has access to the socket.

```yaml
# docker-compose.yml
services:
  netbox2prom:
    image: ghcr.io/hamptik/netbox2prom:latest
    user: "<SYSLOG_NG_UID>:<SYSLOG_NG_GID>"   # must match syslog-ng
    environment:
      ENABLE_SYSLOG: "true"
    volumes:
      - ./config.yml:/etc/netbox2prom/config.yml:ro
      # Config directory — netbox2prom writes here
      - /etc/syslog-ng/conf.d:/etc/syslog-ng/conf.d
      # Control socket — netbox2prom reloads syslog-ng
      - /var/lib/syslog-ng:/var/lib/syslog-ng
```

> **Finding the UID/GID:** On the host running syslog-ng, check with: `id syslog` or `stat -c '%u:%g' /var/lib/syslog-ng/syslog-ng.ctl`

The Docker image includes `syslog-ng-core`, which provides `syslog-ng-ctl` (for reload) and `syslog-ng` (for `--syntax-only` validation).
